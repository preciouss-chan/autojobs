   - UI shows more bullets than PDF exports
   - Preview: 5 exp bullets, 3 project bullets (UI-only display)
   - Export: 1-3 exp bullets, 3 project bullets (in actual PDF)
   - Users see better version in preview than what gets exported

5. TAILORING PRESERVES BULLETS ✓
   - Only summary is tailored
   - Only skills are added/reframed
   - Bullets are INTENTIONALLY NOT modified (by design)
   - Cover letter generated separately
   - Original resume integrity maintained

STAGE 3: PDF EXPORT (Severe Truncation)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
File: /app/api/export/pdf/route.ts

This is WHERE DATA LOSS OCCURS.

Process: fitResumeToOnePage() applies cascading truncation

Page Constraints:
  - Letter size: 8.5" × 11" = 792 points tall
  - Margins: 28 points top + 28 points bottom = 56 points
  - Available content space: 736 points

Truncation Cascade (Lines 207-280):

  STEP 1: Remove Summary (Line 192)
  ─────────────────────────────────
  if (fittedResume.summary) fittedResume.summary = ""
  
  Effect: Professional summary ALWAYS removed
  Loss: 100% of summary content
  Space saved: ~50 points
  
  STEP 2: Reduce Skills (Lines 213, 220, 230)
  ────────────────────────────────────────────
  tools.slice(0, 4)                    // Max 4 (Line 213)
  frameworks_libraries.slice(0, 5)     // Max 5 (Line 220)
  languages.slice(0, 5)                // Max 5 (Line 230)
  
  Effect: Rank skills by evidence, keep best ones
  Loss: ~30% of skills
  Space saved: ~40 points
  
  STEP 3: Reduce Experience Bullets (Lines 243, 254, 265)
  ────────────────────────────────────────────────────────
  First pass:  exp.bullets.slice(0, 3)    // Max 3 bullets
  If too tall: exp.bullets.slice(0, 2)    // Max 2 bullets
  If too tall: exp.bullets.slice(0, 1)    // Max 1 bullet
  
  Effect: Cascade truncation until PDF fits
  Loss: 60-85% of experience bullets
  Space saved: ~100-150 points
  
  (Projects NOT explicitly cascaded, just render with max 3)

Final Result:
  Original Resume Content:
    - Summary: 1-2 lines
    - Projects: 4 × 5-6 bullets = 20-24 bullets
    - Experience: 3 × 4-5 bullets = 12-15 bullets
    - Skills: 20 items (7-8 lang, 6-7 framework, 6-7 tools)
    
  After PDF Export:
    - Summary: 0 lines (REMOVED)
    - Projects: 6-9 bullets total (3 max per project)
    - Experience: 3-5 bullets total (cascaded to 1-3 per entry)
    - Skills: 14 items (5 lang, 5 framework, 4 tools)
  
  DATA RETENTION: 25-30% of original

Data Integrity: 30% ❌

================================================================================
                           CRITICAL ISSUES
================================================================================

ISSUE 1: Summary Always Removed
────────────────────────────────
Severity: HIGH
Location: /app/api/export/pdf/route.ts, Line 192

Problem:
  fittedResume.summary = ""
  
  - Summary is UNCONDITIONALLY removed
  - Even if summary is 1-2 lines (minimal space)
  - Professional summary is important hiring signal
  - May be lost even when there's available space

Recommendation:
  1. Only remove summary if space critical
  2. Consider reducing font size instead of removal
  3. Add option to preserve summary on multi-page export

═══════════════════════════════════════════════════════════════════════════════

ISSUE 2: One-Page Hard Limit
──────────────────────────────
Severity: HIGH
Location: /app/api/export/pdf/route.ts, Lines 95-100

Problem:
  - Hard-coded 1-page PDF limit (757pt available)
  - Forces aggressive truncation
  - No user option for multi-page export
  - Most job applications now accept 2-page resumes

Recommendation:
  1. Allow multi-page export option
  2. Preserve all data on multi-page PDFs
  3. Offer "Single Page" vs "Multi Page" export modes
  4. Show data loss preview before export

═══════════════════════════════════════════════════════════════════════════════

ISSUE 3: No Resume Storage
────────────────────────────
Severity: MEDIUM
Location: /prisma/schema.prisma (missing Resume model)

Problem:
  - Resume not persisted to database
  - Stateless application design
  - User must re-upload on next session
  - No version history
  - No audit trail

Recommendation:
  1. Add Resume model to Prisma schema
  2. Store parsed resume on first upload
  3. Keep version history of tailored resumes
  4. Allow resume re-use without re-parsing

═══════════════════════════════════════════════════════════════════════════════

ISSUE 4: Skill Truncation via Ranking
───────────────────────────────────────
Severity: MEDIUM
Location: /app/api/export/pdf/route.ts, Lines 22-45, 194-205

Problem:
  - rankSkillsByEvidence() removes skills with low frequency
  - Skills are ranked by how often they appear in bullets
  - Important skills may be dropped if rarely mentioned
  - No control over which skills appear

Recommendation:
  1. Preserve all skills by reducing font size instead
  2. Use job-alignment for skill ranking (not just frequency)
  3. Allow user to select which skills to include
  4. Sort by job relevance rather than evidence frequency

ISSUE 6: Bullet Cascade Logic Aggressive
──────────────────────────────────────────
Severity: MEDIUM
Location: /app/api/export/pdf/route.ts, Lines 240-269

Problem:
  - Cascading truncation: 3 → 2 → 1 bullet per entry
  - May result in experience entries with only 1 bullet
  - Experience with single bullet is weak representation
  - No intelligence about which bullets to keep

Recommendation:
  1. Implement intelligent bullet ranking
  2. Preserve strongest 2-3 bullets, not just newest
  3. Use job description context for ranking
  4. Allow multi-page to preserve all bullets

================================================================================
                              CONCLUSION
================================================================================

AutoJobs Resume Parsing: ✅ Excellent
  - Comprehensive extraction from PDFs
  - All data captured in structured format
  - Proper validation and error handling
  - Clean, well-tested implementation

AutoJobs Resume Export: ❌ Problematic
  - Aggressive truncation for single-page constraint
  - 70% data loss typical (only 25-30% retention)
  - Summary always removed
  - No user control over content

AutoJobs Resume Storage: ❌ Missing
  - Stateless by design (no persistence)
  - No resume versioning
  - Requires re-upload each session

Overall Assessment:
  The extraction pipeline is well-designed and comprehensive. However, the
  export layer prioritizes fitting onto a single page over preserving resume
  content. This creates a poor user experience where users upload detailed
  resumes, see them in full in the preview, but export reveals significant
  truncation.

  Recommendation: Allow multi-page export option and preserve all data on
  longer resumes. Implement smart truncation only when truly necessary.

Key Insight:
  Data loss is not a parsing problem - it's a RENDERING problem. The data is
  extracted comprehensively but discarded during PDF generation to fit layout
  constraints.
